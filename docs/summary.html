1. Core Philosophy & Trading Principles

Capital Preservation First. Missing a good trade is fine. Taking a bad trade is not. The bot should happily sit idle all day if nothing is worth deploying capital on.
Cut Losers, Ride Winners. Every system in the bot enforces this:
  • Losers get cut by trailing stops, time limits, and cooldowns
  • Winners are never closed by time limits — only trailing stops manage the exit
  • Exception: planned losses from DCA layering are not "losers" — those are intentional
Hit and Run (Scalping) is the primary mode. Quick in-and-outs on spikes and breakouts. Swing trades are allowed only for "opportunity of a lifetime" events (e.g., BTC drops 20% to major weekly support).
Start Small, DCA In, Ride It Out. Always risk ~$50 to start. Nobody can predict exact bottoms — market makers wick through expected support to grab stop-loss liquidity, then reverse. Instead of getting stopped out, DCA into the wick. Keep adding until $100K notional. When in profit: raise leverage, pull capital out, lock break-even, ride.
Secure 3-5% Daily, Then Let It Ride. The bot operates in two distinct phases each day:
  • Phase 1 — HUNTING (0% to 3%): Be aggressive with profit-taking. Take partial profits early, use tight trailing stops, lock break-evens quickly. Every realized dollar counts toward the daily target. Full position sizing (1.0x aggression).
  • Phase 2 — SECURED (3-5%): Daily profit banked. Slightly reduce new entries (0.8x). Transition to normal trailing behavior.
  • Phase 3 — RIDING (5%+): Let remaining positions run with loose trailing stops (1.5x wider trails). Reduce new entries significantly (0.6x). Chase profits, don't cut them short. Trailing stops protect gains without being greedy.
  • Next day: If positions carry over in profit, secure the new day's 3-5% first (phase 1 again), then switch back to riding mode.

2. How It All Fits Together

Architecture: Hub + Bots

The hub (hub_main.py) is the central brain — it runs Monitor, Analytics, and SignalGenerator in-process, stores all shared state in HubState (in-memory), and serves the dashboard. Trading bots are lightweight executors that consume proposals from the hub via HTTP.

Fast Monitor Loop (~1-2 seconds)

Runs concurrently with the main tick. Handles all time-critical actions:

  1. CLOSE_ALL override — if file exists, close everything immediately (1-2s reaction)
  2. Legendary + reversal close — at 100%+ daily with reversal risk, close all immediately
  3. Trailing stops — close positions that hit trailing/breakeven/initial stops
  4. Liquidation risk — emergency close if margin danger detected
  5. Expired quick trades — close timed-out losers (winners keep riding)
  6. Extreme mover entries — fast entry on large price moves

Main Tick Loop (30-300 seconds, adaptive)

Tick interval adapts: scalp=1s, active=60s, swing=300s, idle=60s. Handles:

  1. Scale in — add to winning positions (WINNERS) or DCA down (PYRAMID)
  2. Lever up — PYRAMID positions that recovered: raise leverage
  3. Partial take — PYRAMID: pull capital off the table after lever-up
  4. Wick scalps — counter-scalp wicks while PYRAMID DCA-s
  5. Process trade queue — proposals from hub (CRITICAL → DAILY → SWING). Advisory, never forced.
  6. Assess intel — reads MarketCondition from hub (Fear & Greed, liquidations, whales, etc.)
  7. Legendary email — at 100%+ daily without reversal: email owner, let it ride
  8. Apply gates — should we trade? (tier, drawdown, intel regime, STOP file)
  9. Validate proposals — per-bot-type validator spot-checks proposals before execution
  10. Hedge check — profitable positions showing reversal? Open counter-position
  11. Log status — PnL, positions, stops, intel summary
  12. Record metrics — tick duration, event loop lag, method timings → Prometheus
  13. Daily reset — at midnight UTC: reset counters, compound. Hub sends one consolidated email.

Reset to clean state

# Stop containers (data is on host, safe)
docker compose down

# Wipe ephemeral state only (keeps DBs intact)
find "$HOST_DATA_DIR" -name "*.json" -o -name "*.lock" | xargs rm -f

# Full wipe (destructive — loses trade history)
rm -rf "$HOST_DATA_DIR"         # all DBs, JSON state
rm -rf "$HOST_LOGS_DIR"         # all log files

On next docker compose up, all directories and files are recreated automatically. Data lives on the host via bind mounts — docker compose down -v does NOT delete it.

5. Manual Controls

How to stop the bot or close trades manually:

ActionHowEffect
Stop new trades touch STOP in the bot directory No new positions will be opened. Existing positions continue riding with their trailing stops.
Close everything touch CLOSE_ALL in the bot directory All open positions are market-closed within 1-2 seconds (fast monitor loop). The file is auto-deleted after execution.
Resume trading rm STOP Normal operation resumes on the next tick.

6. Ideas & Future Plans

Strategy ideas and enhancements under review. Not yet implemented. See docs/new_ideas.md for full discussion notes.

6.1 Volatility Straddle (Simultaneous Long + Short)

Open both a long and short on the same asset simultaneously with high leverage (~300x). Wait for a directional move at market open, then manage exits asymmetrically — the winning side rides while the losing side gets cut early.

Key risk: Fees destroy it at high leverage. Only viable with zero/near-zero fee promotions. Whipsaw at market open can stop both legs.

Status: Under review. Would need a dedicated StraddleStrategy with fee-aware entry filtering.

6.2 Super Scalp (High Volume, $1-per-Trade Target)

Target 500 trades/day at $1 profit each. Volume over size. Partially exists via CompoundMomentumStrategy but not optimized for throughput.

Key risk: 500 round-trips = ~$100 in fees. Exchange rate limits cap throughput.

Status: Under review. Tick loop is now adaptive (1-300s) and fast loop runs every 1-2s, so the infra exists. Needs limit orders and per-trade fee tracking.

6.3 Don't Fight the Trend (3rd Holy Rule)

The Three Holy Rules:
  1. Cut the losers
  2. Ride the winners
  3. Don't fight the trend — non-scalp trades must align with the macro trend

Scalps (< 1 day) can go either direction. Anything longer must align with macro bull/bear trend. Currently only soft directional bias exists, no hard gate.

Status: Highest priority. Would need a MacroTrend classifier (BTC SMA 50/200, TV consensus, DeFiLlama TVL, F&G rolling avg) and hard reject for counter-trend non-scalps.

6.4 AI Background Agent (Log & Trade Data Analysis)

An AI agent running as a separate service that analyzes trade logs, bot logs, and market data to find non-obvious correlations and generate natural-language insights.

Capabilities: Trade pattern analysis, log anomaly detection, strategy parameter recommendations, market regime classification, correlation discovery.

Status: Under review. Would run as an additional Docker service with read-only access to hub.db and the hub API. Scheduled hourly to keep LLM costs low.

6.5 Manual Trade Ideas & Price Alerts

Submit manual trade ideas with pair, entry, SL, and TP via the dashboard. Bot either executes immediately (small amount) or sets a price alarm and auto-executes when the entry price is reached.

Status: Under review. Would need a new dashboard form, alarm/watchlist system, and limit-order-like execution logic.

6.6 AMD — Accumulation Manipulation Distribution

Detects the classic AMD pattern that occurs frequently in crypto. Price accumulates in a tight range (consolidation), then spikes up briefly (manipulation / fake-out), then dumps below the accumulation zone (distribution / true move).

How it works:

  • Accumulation: Identify a range-bound zone (low volatility, tight Bollinger bands, multiple touches of support/resistance)
  • Manipulation: Detect a sudden spike above the range (wick or brief close above) — this is the trap, NOT the entry
  • Distribution: Price drops back into and then BELOW the accumulation zone — this is the signal
  • Entry: Short when price breaks below accumulation zone after the manipulation spike. SL above the manipulation high.
  • Works in reverse for bullish AMD (accumulation, dump below to trap shorts, then rally above).

Status: Not started. Would work as a new pattern in core/patterns/ or a dedicated strategy. Needs range detection + spike classification + breakdown confirmation. High frequency pattern in crypto — very valuable.

6.7 Multi-Timeframe Reversal Signals (Fefe Demeny Method)

Watches 5m, 15m, 1h, 4h, and daily timeframes for reversal confirmations using consecutive candle structure. Each timeframe confirmation strengthens the signal. Inspired by Fefe Demeny's reversal detection method.

How it works (LONG example, SHORT is mirrored):

  • Price is in a downtrend (making lower lows)
  • On the 5m chart: identify a cluster of 3 consecutive candles (a "base")
  • Price drops below that base (new lower low) — this marks the base as a "reversal reference"
  • When price CLOSES back above the 3-candle base on the 5m — minimal signal ("5m reversal confirmed")
  • Now watch 15m: same logic. When 15m confirms — entry-grade signal (SL under the lowest point, start trailing + pyramiding)
  • 1h confirmation strengthens conviction; 4h and daily are macro-level confirmations
  • The more timeframes confirm, the stronger the reversal and the larger the position / wider the stop

Integration: Multi-TF data already available via TradingView client (supports 5m, 15m, 1h, 4h, 1D). Would need a dedicated detector that tracks "reversal bases" per symbol per timeframe, and a scoring system (5m = 1pt, 15m = 2pt, 1h = 3pt, 4h = 4pt, daily = 5pt).

Status: Not started. High potential — multi-TF alignment is one of the strongest reversal confirmation methods. Could work as a standalone strategy or as a signal booster for existing strategies.

7.6 Tiered Daily Targets & Manual Override

  1. Capital is safe (never risk what you have)
  2. Secure 3-5% daily (aggressive profit-taking until banked)
  3. Once secured, ride with loose stops (let winners run)
  4. Let it grow to 20%, 50% (reduce activity progressively)
  5. At 100%: close all if reversal risk detected, or email owner explaining why we ride
